  ROLE GLOSSARY
  ─────────────────────────────────────────────────────────────────────────────────────────────────────────────────────────
  ADMIN    : Full access to all features. Can submit and approve requests from different users.
  MAKER    : Submits data change requests (create/update/delete). Cannot approve any request.
  CHECKER  : Reviews and approves or rejects Maker submissions. Cannot submit change requests.
             Note: A Checker may not approve a request they submitted themselves.
  VIEWER   : Read-only. Can view Users, Dashboard, and Audit Trail. Cannot access Pending Actions.

  Pending Actions   Reject action                ✓*      ✗       ✓*       ✗
  Pending Actions   Cancel own action            ✓*      ✓*      ✗        ✗
  Pending Actions   Cancel others' action        ✗       ✗       ✗        ✗

  ✓* = Approve/Reject: allowed only when the requester is a different user (maker ≠ checker).
  ✓* = Cancel: allowed only by the original maker of that specific action.

=============================================================================================================================
  NOTES
=============================================================================================================================

  1. All "403 Forbidden" responses are enforced at the backend via @PreAuthorize on each controller
     method. The frontend also hides buttons and redirects for UI-level enforcement, but the API
     remains the authoritative gate.

  2. The "maker ≠ checker" constraint (M.19) is enforced at the service layer, not the security
     layer. A CHECKER who calls approve on their own submission receives a business error (400/422),
     not a 403.

  3. The Cancel endpoint (M.22–M.23) has a two-layer check:
       • Security layer: only ADMIN and MAKER roles can call the endpoint (others get 403).
       • Service layer: among ADMIN/MAKER callers, only the original maker passes (others get 400).

  4. Tests M.6, M.7, M.16, M.18, M.20, M.22 require MAKER and CHECKER test users to be active.
     These are conditionally skipped in the E2E suite if those users are unavailable (see
     global-setup.js and the maker-checker seed constraint).
